# Adding a Module

This guide walks through adding a new inspection module to oxi-nixinfra using the
`nix_module!` macro. We'll use the `Sysctl` module as the primary example — it's
the simplest case. `Environment` shows how to add per-method arguments.
`SystemInfo` shows how to mix macro-generated and hand-written methods.

## How the macro works

`nix_module!` generates four things from a single declaration:

1. A `#[pyclass(frozen)]` sync struct (e.g., `Sysctl`)
2. A `#[pyclass(frozen)]` async struct (e.g., `AsyncSysctl`)
3. A `pub fn new(inner, ...)` constructor on both structs
4. `#[pymethods]` blocks on both structs — sync methods call `wrap_sync`,
   async methods call `future_into_py`

You write only the `_impl` async core functions. The macro handles all the
boilerplate.

## Step 1: Create the module file with `_impl` functions

Create `src/modules/sysctl.rs`. Write one `pub async fn` per operation, taking
`&HostInner` first, then any struct fields (passed automatically by the macro),
then any per-call arguments.

```rust
use crate::backend::BackendError;
use crate::host::HostInner;

// ---------------------------------------------------------------------------
// Layer 1 — Async core functions
// ---------------------------------------------------------------------------

pub async fn value_impl(inner: &HostInner, key: &str) -> Result<String, BackendError> {
    let out = inner.execute("sysctl", &["-n", key]).await?;
    if out.rc == 0 {
        let stdout = String::from_utf8_lossy(&out.stdout);
        Ok(stdout.trim().to_owned())
    } else {
        Err(BackendError::Execution(format!(
            "sysctl failed for '{}': {}",
            key,
            String::from_utf8_lossy(&out.stderr)
        )))
    }
}

pub async fn exists_impl(inner: &HostInner, key: &str) -> Result<bool, BackendError> {
    let out = inner.execute("sysctl", &["-n", key]).await?;
    Ok(out.rc == 0)
}

// ---------------------------------------------------------------------------
// Layers 2 + 3 — Generated by macro
// ---------------------------------------------------------------------------

oxi_nixinfra_macros::nix_module! {
    Sysctl {
        fields { key: String },
        new(key: &str) {
            key: key.to_owned(),
        },
        methods {
            fn value() -> String => value_impl;
            fn exists() -> bool => exists_impl;
        }
    }
}
```

### `nix_module!` syntax

```
nix_module! {
    StructName {
        fields { field_name: FieldType, ... },
        new(param: ParamType, ...) {
            field_name: init_expr,
            ...
        },
        methods {
            fn method_name(extra_arg: ArgType, ...) -> ReturnType => impl_fn_name;
            ...
        }
    }
}
```

- **`fields`** — stored on the struct, passed as `&self.field` to every `_impl`
  call automatically. No fields? Write `fields {}`.
- **`new(...)`** — constructor parameters that become `Host::sysctl(key)` on the
  Python side. The `inner: Arc<HostInner>` argument is always prepended by the
  macro — do not list it here.
- **`methods`** — extra per-call arguments only. The macro prepends `&inner` and
  all `&self.field` refs automatically. `&str` params are cloned to `String`
  before the async move block so they cross the `'static` boundary.

The macro generates `AsyncSysctl` alongside `Sysctl` — both from the same
declaration.

## Step 2: Add to `register_modules!` in `host.rs`

Open `src/host.rs`. At the bottom of the file there is a `register_modules!`
invocation. Add one line for the new module:

```rust
oxi_nixinfra_macros::register_modules! {
    Host, AsyncHost;
    // ... existing entries ...
    sysctl(key: &str) -> sysctl::Sysctl,
}
```

The entry format is `method_name(params) -> module::StructName`. This generates
`Host::sysctl(key)` returning `Sysctl` and `AsyncHost::sysctl(key)` returning
`AsyncSysctl`.

## Step 3: Add to `src/modules/mod.rs`

```rust
pub mod sysctl;
```

## Step 4: Register pyclasses in `src/lib.rs`

```rust
m.add_class::<modules::sysctl::Sysctl>()?;
m.add_class::<modules::sysctl::AsyncSysctl>()?;
```

Both the sync and async struct must be registered.

## Step 5: Add Python re-exports in `__init__.py`

Open `python/oxi_nixinfra/__init__.py`. Add both names to the import and to
`__all__`:

```python
from oxi_nixinfra._oxi_nixinfra import (
    # ... existing imports ...
    Sysctl,
    AsyncSysctl,
)

__all__ = [
    # ... existing entries ...
    "Sysctl",
    "AsyncSysctl",
]
```

## Example: Environment — extra method arguments

`Sysctl` stores its key as a struct field, so the methods take no extra
arguments. `Environment` takes no constructor argument at all — instead, the
variable name is passed at call time:

```rust
oxi_nixinfra_macros::nix_module! {
    Environment {
        fields {},
        new() {},
        methods {
            fn get(name: &str) -> Option<String> => get_impl;
            fn exists(name: &str) -> bool => exists_impl;
        }
    }
}
```

The corresponding `_impl` signatures:

```rust
pub async fn get_impl(inner: &HostInner, name: &str) -> Result<Option<String>, BackendError>
pub async fn exists_impl(inner: &HostInner, name: &str) -> Result<bool, BackendError>
```

Because `fields {}` is empty, the macro passes only `&inner` before the
per-method args. The Python call looks like `host.environment().get("PATH")`.

In `register_modules!` this entry has no constructor parameters:

```rust
environment() -> environment::Environment,
```

## Example: SystemInfo — partial macro, partial manual methods

When a method needs to extract sub-fields from a richer return type, declare
only the straightforward methods in `nix_module!` and add additional
`#[pymethods]` blocks manually. The macro-generated struct is open for extension
— PyO3 allows multiple `#[pymethods]` blocks on the same struct.

```rust
oxi_nixinfra_macros::nix_module! {
    SystemInfo {
        fields {},
        new() {},
        methods {
            fn nixos_version() -> String => nixos_version_impl;
            fn system_profile() -> String => system_profile_impl;
            fn generation_count() -> i32 => generation_count_impl;
            fn specialisations() -> Vec<String> => specialisations_impl;
        }
    }
}

// Methods that don't follow the standard pattern (extract fields from BootInfo)
#[pyo3::prelude::pymethods]
impl SystemInfo {
    fn kernel_version(&self) -> pyo3::prelude::PyResult<String> {
        let info = crate::helpers::wrap_sync(&self.inner, boot_info_impl(&self.inner))?;
        Ok(info.kernel_version)
    }

    fn arch(&self) -> pyo3::prelude::PyResult<String> {
        let info = crate::helpers::wrap_sync(&self.inner, boot_info_impl(&self.inner))?;
        Ok(info.arch)
    }
    // ...
}

#[pyo3::prelude::pymethods]
impl AsyncSystemInfo {
    fn kernel_version<'py>(
        &self,
        py: pyo3::prelude::Python<'py>,
    ) -> pyo3::prelude::PyResult<pyo3::prelude::Bound<'py, pyo3::prelude::PyAny>> {
        let inner = self.inner.clone();
        pyo3_async_runtimes::tokio::future_into_py(py, async move {
            let info = boot_info_impl(&inner)
                .await
                .map_err(crate::helpers::backend_err_to_py)?;
            Ok(info.kernel_version)
        })
    }
    // ...
}
```

Use the fully-qualified paths (`pyo3::prelude::*`,
`crate::helpers::wrap_sync`, etc.) in manual `#[pymethods]` blocks — the module
file does not have a top-level `use pyo3::prelude::*` import because the macro
handles that internally for generated code.

The key rule: if the method signature doesn't fit
`fn foo_impl(&HostInner, /* fields */, /* extra args */) -> Result<T, BackendError>`,
write it by hand. Common reasons:

- The `_impl` returns a struct and you need to destructure it (SystemInfo `boot_info_impl`).
- The method requires multi-step async logic before calling `_impl` (User `resolve_name`).
- The return type or argument shape can't be expressed in the macro's grammar.

## Step 6: Testing with `MockBackend`

`MockBackend` lives in `src/backend/mock.rs` and is available only in
`#[cfg(test)]`. It replays a fixed queue of `RawOutput` values in FIFO order and
records every `(program, args)` call.

Standard test structure for a module:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use crate::backend::mock::MockBackend;
    use crate::command::RawOutput;

    fn make_inner(responses: Vec<RawOutput>) -> HostInner {
        HostInner {
            backend: Box::new(MockBackend::new(responses)),
            runtime: tokio::runtime::Runtime::new().unwrap(),
            connection_string: "mock://".to_owned(),
        }
    }

    #[test]
    fn test_value_returns_trimmed() {
        let inner = make_inner(vec![RawOutput {
            rc: 0,
            stdout: b"1\n".to_vec(),
            stderr: vec![],
        }]);
        assert_eq!(
            inner
                .runtime
                .block_on(value_impl(&inner, "net.ipv4.ip_forward"))
                .unwrap(),
            "1",
            "trailing newline must be stripped",
        );
    }

    #[test]
    fn test_value_error_on_unknown_key() {
        let inner = make_inner(vec![RawOutput {
            rc: 1,
            stdout: vec![],
            stderr: b"sysctl: cannot stat /proc/sys/nonexistent: No such file or directory\n"
                .to_vec(),
        }]);
        let err = inner
            .runtime
            .block_on(value_impl(&inner, "nonexistent"))
            .unwrap_err();
        assert!(
            err.to_string().contains("sysctl failed"),
            "error message must identify the failure source",
        );
    }
}
```

Test the `_impl` functions directly — they are plain async functions and don't
need PyO3 or a real host. Call them with `runtime.block_on(...)`.

Every `assert!` must carry a message. oxitest runs with `strict = "abort"` and
bare asserts are violations. The message explains *why* the assertion matters,
not *what* it checks (oxitest already shows expected vs. actual).

For a Python integration test using the real host, see
[Testing](testing.md).
